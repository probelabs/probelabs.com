<section class="section section-pillars">
  <div class="container">
    <div class="section-header">
      <span class="section-label">The Problem</span>
      <h2>What We Replace</h2>
      <p class="section-desc">Common platform headaches that slow down your engineering organization.</p>
    </div>
    <div class="pillars-grid">
      <div class="pillar-card">
        <div class="pillar-number">01</div>
        <h3>Ad-hoc Agent Scripts</h3>
        <p>Scripts that only work "in one place" - your local machine, that one CI job, or a specific server. When the environment changes, everything breaks.</p>
        <ul class="pillar-benefits">
          <li>Environment-specific hardcoding</li>
          <li>Breaks on deployment</li>
          <li>No portability</li>
        </ul>
      </div>
      <div class="pillar-card">
        <div class="pillar-number">02</div>
        <h3>Copy-Pasted Logic</h3>
        <p>The same AI workflow logic duplicated across repos and teams. Updates require finding and changing code in multiple places.</p>
        <ul class="pillar-benefits">
          <li>Diverging implementations</li>
          <li>Bug fixes applied inconsistently</li>
          <li>No central ownership</li>
        </ul>
      </div>
      <div class="pillar-card">
        <div class="pillar-number">03</div>
        <h3>Black-Box AI Behavior</h3>
        <p>No traces, no ownership, and no auditability. When something goes wrong, you have no idea what the agent did or why.</p>
        <ul class="pillar-benefits">
          <li>Impossible to debug</li>
          <li>No compliance story</li>
          <li>Security blind spots</li>
        </ul>
      </div>
    </div>
  </div>

<!-- FAQ Section -->
<section class="section section-faq">
  <div class="container">
    <div class="section-header">
      <span class="section-label">FAQ</span>
      <h2>Common Questions from Platform Teams</h2>
    </div>
    <div class="faq-grid">
      <div class="faq-item">
        <h3>How does this integrate with our existing CI/CD?</h3>
        <p>Visor workflows run as steps in your existing pipelines - GitHub Actions, GitLab CI, Jenkins, whatever you use. They're just another step, with full observability hooks into your monitoring stack.</p>
      </div>
      <div class="faq-item">
        <h3>Can we run this fully on-prem?</h3>
        <p>Yes. Probe runs entirely locally - retrieval runs locally and you control what context is sent to the model. Visor can run self-hosted or air-gapped. You choose which LLM to connect to, including self-hosted options.</p>
      </div>
      <div class="faq-item">
        <h3>How do teams customize shared workflows?</h3>
        <p>Workflow blocks expose parameters. Teams import blocks and set parameters for their context - different repos, different Slack channels, different approval thresholds. Same logic, team-specific config.</p>
      </div>
      <div class="faq-item">
        <h3>What observability do we get?</h3>
        <p>Every workflow step emits OpenTelemetry traces. You see latency, token usage, LLM calls, tool invocations, and outputs. Export to Datadog, Grafana, or any OTEL-compatible backend. Debug AI workflows like any other system.</p>
      </div>
