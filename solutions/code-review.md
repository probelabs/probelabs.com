          <p class="capability-lead">Reviews that understand your entire system, not just the diff. Catch issues that span repos, services, and dependencies.</p>
          <ul class="capability-questions">
            <li>"Does this change break the API contract?"</li>

It knows when a change in one service breaks consumers in another repo. It catches architectural violations that no single file diff could reveal.</p>

<section class="section section-faq">
  <div class="container">
    <div class="section-header">
      <span class="section-label">Common Questions</span>
      <h2>What teams ask us about AI code review</h2>
    </div>
    <div class="faq-grid">
      <div class="faq-item">
        <h3>How is this different from GitHub Copilot or linters?</h3>
        <p>Copilot suggests code. Linters check style. Neither understands your architecture. They don't know that this change breaks an API contract, or that this pattern violates your team's standards, or that three other services depend on the interface you're modifying.</p>
        <p>Probe understands your entire system -- dependencies, contracts, patterns, and historical decisions. It reviews code the way a senior engineer would, with full context.</p>
      </div>
      <div class="faq-item">
        <h3>Does this replace human code review?</h3>
        <p>No. It replaces the first pass -- the mechanical checking for security issues, missing tests, style violations, and obvious bugs. Human reviewers focus on what they're good at: design decisions, business logic, and architectural trade-offs. Not hunting for missing error checks.</p>
      </div>
      <div class="faq-item">
        <h3>Can we customize the review rules?</h3>
        <p>Yes, extensively. Define rules per repo, per team, per language. Different standards for core services vs. experiments. Agency working with multiple clients? Different rules per client. Rules live as code, versioned alongside your codebase, updated through PRs.</p>
      </div>
      <div class="faq-item">
        <h3>What about false positives?</h3>
        <p>Every automated review system has false positives. Probe minimizes them by understanding context -- it knows the difference between a security vulnerability and a safe pattern. When it does flag something incorrectly, you can tune the rule or suppress that check. Over time, your review quality improves as the rules learn from your feedback.</p>
      </div>
    </div>
